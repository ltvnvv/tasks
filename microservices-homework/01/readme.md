### Задание 1

### Сравнительная таблица решений:

| **Решение**              | **Маршрутизация запросов**     | **Проверка аутентификации**       | **Терминация HTTPS**             |
|--------------------------|--------------------------------|-----------------------------------|----------------------------------|
| **Kong**                 | Поддержка динамической маршрутизации на основе конфигурации (интерфейс Admin API) | OAuth2, API Key, JWT, LDAP | Есть |
| **NGINX API Gateway**     | Конфигурация маршрутов через NGINX конфиг. Поддержка динамической маршрутизации | JWT, Basic Auth, OAuth2 | Есть |
| **Traefik**               | Автоматическая маршрутизация, интеграция с Kubernetes и Docker | JWT, Basic Auth, OAuth2 | Есть |
| **HAProxy**               | Поддерживает сложную маршрутизацию на основе различных условий, таких как URL-пути, хедеры, методы запросов | Basic Auth | Есть |

### Выбор решения:

На основе сравнения можно сделать выводы:

- **Kong** будет отличным выбором для организаций, требующих гибкости и масштабируемости с множеством интеграций, а также возможности быстро разворачивать решения.
- **Traefik** идеально подойдет для пользователей, использующих Docker и Kubernetes и нуждающихся в простоте развертывания и настройки.
- **NGINX API Gateway** лучше всего подойдет для компаний с высокими требованиями к производительности и возможностью настроить каждую деталь работы шлюза.
- **HAProxy** — это высокопроизводительный и очень надежный балансировщик нагрузки, который также может выполнять функции API Gateway, если его правильно настроить.

### Вывод:

Если система использует контейнеризацию **Traefik** — это лучший выбор.

### Задание 2

### Сравнительная таблица брокеров сообщений

| Характеристика                         | **Apache Kafka**                   | **RabbitMQ**                     | **ActiveMQ**                     | **Apache Pulsar**                 |
|----------------------------------------|------------------------------------|----------------------------------|----------------------------------|-----------------------------------|
| **Поддержка кластеризации**            | Да, n-копии и разделы, высокая отказоустойчивость | Да, кластеризация с возможностью репликации | Да, поддержка кластеров и репликации | Да, встроенная кластеризация и репликация |
| **Хранение сообщений на диске**       | Да, хранение сообщений на диске для обеспечения durability | Да, поддержка хранения сообщений на диске (с настройками TTL) | Да, поддержка хранения на диске | Да, на диске с возможностью политики хранения |
| **Высокая скорость работы**            | **1-2 млн TPS** (с оптимизацией)  | **100k - 200k TPS** (с оптимизацией) | **50k - 100k TPS** (в зависимости от конфигурации) | **1-2 млн TPS** (с оптимизацией) |

| **Простота эксплуатации**              | Сложное управление и настройка, но мощные инструменты мониторинга | Прост в настройке, развитая документация, но может быть сложен при масштабировании | Прост в настройке, но имеет ограниченную производительность | Сложное управление, но хорошие возможности для мониторинга и масштабирования |

### Обоснованный выбор:

- **Apache Kafka**: с производительностью до **1-2 млн TPS** (в зависимости от конфигурации и оптимизации) является отличным выбором для обработки больших объемов сообщений и обеспечения отказоустойчивости. Она поддерживает масштабирование и надежное хранение данных на диске.
- **RabbitMQ**: с производительностью от **100k до 200k TPS** подходит для меньших и средних нагрузок, но может быть сложнее масштабировать на очень высоких объемах данных.
- **ActiveMQ**: с производительностью **50k - 100k TPS** подходит для средних нагрузок, но имеет ограниченную масштабируемость по сравнению с Kafka и Pulsar.
- **Apache Pulsar**: также обладает высокой производительностью, до **1-2 млн TPS**, и хорошо масштабируется при большом количестве данных и подписчиков.

### Вывод:

**Apache Kafka** является предпочтительным выбором для систем, требующих высокой производительности и масштабируемости. Kafka идеально подходит для обработки больших потоков данных.
